# ListIQ Pricing Model

## Overview

XGBoost regression model predicting `final_sale_price` for resale items across eBay, Poshmark, and Depop. Trained on 4,735 rows (80% of 5,919 unified dataset), tested on 1,184 rows.

- **Target:** `log1p(final_sale_price)` — predictions inverted with `expm1`
- **Architecture:** One unified model with `platform` as a feature
- **Training notebook:** `notebooks/02_pricing_model.ipynb`

## Files

| File | Description |
|------|-------------|
| `pricing_model.joblib` | Serialized sklearn Pipeline (preprocessor + XGBoost) |
| `feature_config.joblib` | Brand lists, condition map, feature column names |

## Loading the model


## Caveats

1. **Depop predictions are listed-price proxies**, not sold-price predictions. Depop does not expose sold-price history.
2. **Luxury items (>$200)** have high MAE ($339) due to variance and thin data.
3. **Budget items** have high MedAPE (124%) because small absolute errors produce large percentage errors on low-price items.
4. **595/712 brands** have <5 items and are treated as "unknown." Brand signal comes from top 30 brands + tier classification.
5. **No photo features, no temporal features.** Visual condition and seasonality are not captured.

## Platform Router (`router.py`)

Turns a single item description into a ranked recommendation: which platform to list on, at what price, with what expected economics, and whether it's worth listing at all.

### Usage

```python
from models.router import recommend_listing

result = recommend_listing({
    "category": "denim jacket",
    "brand": "Levi's",
    "condition": "Like New",
    "size": "M",
    "color": "blue",
    "estimated_retail": 89.99,
})
```

Returns a dict matching the locked frontend JSON contract with `item`, `recommendations` (3 platforms ranked), and `worth_it` verdict.

### How it works

1. **Price prediction** — Calls the XGBoost pricing model per platform
2. **Price tiers** — `fast_sale` / `balanced` / `max_revenue` derived from model prediction × data-driven percentile ratios (p25/p50 and p75/p50 per category×platform)
3. **Sell-through estimation** — Poshmark days-to-sale from real data; eBay and Depop estimated via platform multipliers
4. **Net profit** — Price × (1 − fee) − shipping
5. **Fit score** — `(net_profit × sell_probability) / sqrt(days_to_sale)`, normalized to 0–10 per item
6. **Worth It** — Based on effective hourly rate (net profit ÷ 35 min seller time)

### Assumptions (auditable constants in `router.py`)

| Assumption | Value | Source |
|------------|-------|--------|
| eBay fee | 13% | Platform policy |
| Poshmark fee | 20% | Platform policy |
| Depop fee | 10% | Platform policy |
| Shipping | $5.99 flat | Estimate |
| Seller time | 35 min/item | Estimate |
| eBay velocity | 0.75× Poshmark | Assumption — larger buyer pool, stronger search |
| Depop velocity | 1.1× Poshmark | Assumption — narrower audience |
| Worth It threshold | ≥$20/hr | Team decision |
| Marginal threshold | $10–20/hr | Team decision |
| Not worth it | <$10/hr | Team decision |

### Price spread ratios

Generated by `scripts/compute_router_constants.py` from `all_platforms.csv`. Re-run after data refresh:

```bash
python scripts/compute_router_constants.py
```

### Evaluation results

- **Schema compliance:** 5/5 test items pass validation
- **Routing accuracy:** 7/8 categories match EDA ground truth (blazer is the sole disagreement — it's the only category with no significant cross-platform price difference, so routing is ambiguous)

### Caveats

1. **Depop predictions are listed-price proxies**, not sold-price predictions
2. **Sell-through estimates for eBay/Depop are assumptions**, not measured from data — documented as named constants
3. **Luxury items (>$200)** have high model uncertainty (MAE $339)
4. **Price tiers use percentile ratios from current dataset** — ratios go stale if data changes without re-running `compute_router_constants.py`
5. **Input fields `size` and `color`** are used for title synthesis only; they are not direct model features
